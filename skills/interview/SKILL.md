# Interview

## 재사용 경로

이 문서는 요구사항 구체화와 ADR 초안 정리가 필요할 때 재사용하는 인터뷰 규칙입니다.

공통 규칙은 `CLAUDE.md`, `docs/guides/README.md`, `docs/adr/CLAUDE.md`, 그리고 관련 command hook 결과를 기준으로 해석하세요.

## 목표

- 기능이나 문제를 구현 가능한 수준으로 구체화합니다.
- 사용자에게 불필요한 질문은 줄이고, 결정이 필요한 지점만 드러냅니다.
- 정보가 충분하면 ADR 초안 방향과 PLANNED 반영 포인트까지 정리합니다.

## 인터뷰 단계

### 1. 문제 정의

- 현재 상황은 무엇인가
- 어떤 문제가 반복되는가
- 지금 해결해야 하는 이유는 무엇인가

### 2. 기술 요구사항

- 성능/확장성 요구사항
- 보안/권한 요구사항
- 기존 시스템과의 호환성
- 배포/운영 제약

### 3. 사용자/비즈니스 요구사항

- 누가 사용하는가
- 성공 기준은 무엇인가
- 마감/우선순위/비용 제약은 무엇인가

### 4. 경계 조건

- 실패 시 처리 방식
- 동시성/일관성 문제
- 롤백 또는 단계적 출시 필요 여부

### 5. Trade-off

- 단순성 vs 유연성
- 속도 vs 완성도
- 비용 vs 운영 복잡도

## 질문 규칙

- 한 번에 너무 많은 질문을 던지지 않습니다.
- 이미 답한 내용은 반복하지 않습니다.
- 구현 방향을 좌우하는 질문부터 우선합니다.
- 사용자에게 선택을 요구할 때는 각 선택의 trade-off를 같이 설명합니다.

## 산출물

인터뷰가 끝나면 다음을 남깁니다.

1. 확정된 요구사항 요약
2. 남은 결정 사항
3. 추천 구현 방향
4. 필요 시 ADR 초안 방향
5. 필요 시 architecture 문서의 PLANNED 반영 포인트

## 문서 규칙

- ADR 초안을 만든다면 `docs/adr/ADR-0000-템플릿.md`와 `docs/adr/README.md`를 기준으로 합니다.
- PLANNED 표기는 `docs/guides/README.md`와 `docs/architecture/README.md` 규칙을 따릅니다.
- 정보가 부족하면 문서를 억지로 만들지 말고 미결정 사항을 분리해 남깁니다.
